# アルゴリズム仕様

## 目的

本ドキュメントでは、Hit & Blow Solver の主要アルゴリズムについて説明する。

各関数の実装内容ではなく、どのような考え方で処理を行っているかをまとめる。

---

# 候補生成

対象関数

- generateCandidates()
- generateCandidatesNoDuplicate()
- generateCandidatesAllowDuplicate()

## 処理概要

ゲーム開始時に、ルールに応じた全候補を生成する。

現在は重複なしルールのみ実装している。

### 重複なし

各桁に未使用の色のみを配置する。

```
開始
 ↓
1桁目を決定
 ↓
2桁目を決定
 ↓
3桁目を決定
 ↓
4桁目を決定
 ↓
候補一覧へ追加
```
再帰処理を利用して全パターンを生成する。

6色・4桁・重複なしの場合、

360通り

の候補が生成される。

---

# Hit・Blow判定

対象関数

- judgeHitBlow()

## アルゴリズム

Hit を先に数え、その後 Blow を数える。

```
guess
↓
Hit判定
↓
Hit以外だけ残す
↓
Blow判定
↓
終了
```

### Hit

同じ位置・同じ色なら Hit とする。

### Blow

Hit にならなかった色だけを比較する。

一致した色は answerRemain から削除することで、
同じ色を二重に数えないようにしている。

---

# 候補絞り込み

対象関数

- filterCandidates()

## 処理概要

候補を1件ずつ評価し、
ゲームで返ってきた Hit・Blow と一致する候補だけを残す。

```
候補一覧
   │
   ▼
judgeHitBlow()
   │
   ▼
  一致？
 │      │
YES    NO
 │      │
残す   捨てる
```

全候補に対してこの処理を行う。

---

# 候補再構築

対象関数

- rebuildCandidates()

## 処理概要

履歴を先頭から順番に適用し、
毎回候補を再計算する。

```
全候補
   │
   ▼
1回目
   │
   ▼
2回目
   │
   ▼
3回目
   │
   ▼
現在候補
```

履歴修正が行われた場合も、
最初から再計算することで整合性を保っている。

---

# ヒートマップ

対象関数

- calculateHeatmap()

## 処理概要

現在候補の各桁について、
各色が出現する割合を算出する。

```
候補一覧
   │
   ▼
1桁目集計
2桁目集計
3桁目集計
4桁目集計
   │
   ▼
割合計算
```

割合は

出現回数 ÷ 候補数 × 100

で算出する。

候補数が0の場合は 0% とする。

---

# 入力値チェック

対象関数

- validateResult()

## チェック内容

以下の場合は入力エラーとする。

- Hit < 0
- Blow < 0
- Hit > 桁数
- Blow > 桁数
- Hit + Blow > 桁数

また、ゲームルール上あり得ない組み合わせについても、
候補が1件も存在しない場合は入力エラーとして扱う。

---

# 設計思想

本ツールでは、

- 判定ロジック
- 候補生成
- 描画処理

を分離して実装している。

ロジックは UI に依存せず、
UI は gameState の状態のみを参照する構成としている。

このため、

- 将来的なルール追加
- UI変更
- ロジック改善

を相互に影響を少なく保守できる設計となっている。
